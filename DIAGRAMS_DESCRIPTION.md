# 1. Диаграмма вариантов использования (Use Case)

## 1.1 Акторы (Actors)

### Первичные акторы (Primary Actors):

| Актор | Описание | Роль в системе |
|-------|----------|----------------|
| **Гость** | Неавторизованный пользователь | Может только войти в систему или зарегистрироваться |
| **Оператор** | Сотрудник, обслуживающий автоматы | Просмотр статусов, мониторинг, базовые отчёты |
| **Инженер** | Технический специалист | Управление обслуживанием, диагностика, ремонт |
| **Менеджер** | Управляющий сетью автоматов | Полный доступ к аналитике, управление персоналом |
| **Администратор** | Системный администратор | Полный доступ ко всем функциям, управление пользователями |
| **Франчайзи** | Партнёр-арендатор | Просмотр своих автоматов, договоры аренды |

### Вторичные акторы (Secondary Actors):

| Актор | Описание |
|-------|----------|
| **API Сервер** | Внешняя система (VendingMachineAPI), обрабатывающая запросы |
| **База данных** | PostgreSQL, хранящая все данные системы |
| **Модем/Телеметрия** | Устройство связи торгового автомата |

#### UC-01: Войти в систему
- **Актор**: Гость
- **Предусловие**: Пользователь не авторизован, имеет учётную запись
- **Основной сценарий**:
  1. Гость открывает страницу входа
  2. Гость вводит email
  3. Гость вводит пароль
  4. Система проверяет учётные данные
  5. Система генерирует JWT токен
  6. Система обновляет время последнего входа
  7. Система перенаправляет на главную страницу
- **Альтернативный сценарий**:
  - 4a. Неверный email: Система показывает ошибку "Неверные учётные данные"
  - 4b. Неверный пароль: Система показывает ошибку "Неверные учётные данные"
  - 4c. Аккаунт заблокирован: Система показывает ошибку "Аккаунт неактивен"
- **Постусловие**: Пользователь авторизован, токен сохранён

#### UC-02: Зарегистрироваться
- **Актор**: Гость
- **Предусловие**: Пользователь не имеет учётной записи
- **Основной сценарий**:
  1. Гость открывает страницу регистрации
  2. Гость вводит email
  3. Гость вводит пароль (мин. 8 символов, цифры, спецсимволы)
  4. Гость вводит полное имя
  5. Система валидирует данные
  6. Система проверяет уникальность email
  7. Система хеширует пароль (BCrypt)
  8. Система создаёт учётную запись
  9. Система показывает сообщение об успехе
- **Альтернативный сценарий**:
  - 6a. Email занят: Система показывает ошибку "Пользователь уже существует"
  - 5a. Невалидный пароль: Система показывает требования к паролю
- **Постусловие**: Учётная запись создана

#### UC-03: Выйти из системы
- **Актор**: Авторизованный пользователь (любая роль)
- **Предусловие**: Пользователь авторизован
- **Основной сценарий**:
  1. Пользователь нажимает "Выход"
  2. Система очищает JWT токен
  3. Система очищает данные сессии
  4. Система перенаправляет на страницу входа
- **Постусловие**: Пользователь разлогинен

#### UC-04: Просмотреть Dashboard
- **Акторы**: Оператор, Инженер, Менеджер, Администратор
- **Предусловие**: Пользователь авторизован
- **Основной сценарий**:
  1. Пользователь переходит на главную страницу
  2. Система загружает данные Dashboard с API
  3. Система отображает виджет эффективности сети (gauge)
  4. Система отображает круговую диаграмму статуса сети
  5. Система отображает сводку (продажи, инкассация, обслуживание)
  6. Система отображает график динамики продаж
  7. Система отображает ленту новостей
- **Включает**: Загрузить эффективность сети, Загрузить статус сети, Загрузить динамику продаж, Загрузить новости
- **Постусловие**: Dashboard отображён с актуальными данными

#### UC-05: Просмотреть монитор торговых автоматов
- **Акторы**: Оператор, Инженер, Менеджер, Администратор
- **Предусловие**: Пользователь авторизован
- **Основной сценарий**:
  1. Пользователь переходит в раздел "Монитор ТА"
  2. Система загружает список торговых автоматов
  3. Система отображает таблицу с данными:
     - Номер, название, адрес
     - Статус связи (онлайн/офлайн)
     - Уровень загрузки ингредиентов
     - Денежные средства (купюры, монеты, сдача)
     - Статус оборудования
     - Последняя активность
  4. Пользователь может применить фильтры
  5. Пользователь может отсортировать данные
  6. Пользователь может экспортировать в CSV
- **Расширяет**: Фильтровать по статусу, Фильтровать по подключению, Сортировать данные, Экспортировать в CSV
- **Постусловие**: Таблица мониторинга отображена

#### UC-06: Просмотреть детали автомата
- **Акторы**: Все авторизованные пользователи
- **Предусловие**: Пользователь авторизован, автомат существует
- **Основной сценарий**:
  1. Пользователь выбирает автомат из списка
  2. Система загружает детальную информацию
  3. Система отображает:
     - Основные данные (серийный номер, модель, локация)
     - Технические параметры
     - Историю обслуживания
     - Статистику продаж
- **Постусловие**: Детальная информация отображена

#### UC-08: Просмотреть список торговых автоматов
- **Акторы**: Менеджер, Администратор
- **Предусловие**: Пользователь авторизован с соответствующей ролью
- **Основной сценарий**:
  1. Пользователь переходит в "Администрирование" → "Торговые автоматы"
  2. Система загружает список ТА с пагинацией (10 на странице)
  3. Система отображает таблицу:
     - Серийный номер, название, модель
     - Статус, расположение
     - Доход, дата установки
  4. Пользователь может переключить вид (таблица/плитки)
  5. Пользователь может искать по названию
  6. Пользователь может переходить между страницами
- **Постусловие**: Список ТА отображён

#### UC-09: Добавить торговый автомат
- **Акторы**: Менеджер, Администратор
- **Предусловие**: Пользователь авторизован с соответствующей ролью
- **Основной сценарий**:
  1. Пользователь нажимает "Добавить"
  2. Система открывает диалог создания
  3. Пользователь заполняет обязательные поля:
     - Серийный номер (уникальный)
     - Название
     - Модель
     - Расположение
     - Дата установки
  4. Пользователь заполняет опциональные поля
  5. Пользователь нажимает "Сохранить"
  6. Система валидирует данные
  7. Система создаёт запись в БД
  8. Система обновляет список
  9. Система показывает уведомление об успехе
- **Альтернативный сценарий**:
  - 6a. Серийный номер не уникален: Ошибка "Автомат с таким серийным номером уже существует"
- **Постусловие**: Торговый автомат создан

#### UC-10: Редактировать торговый автомат
- **Акторы**: Менеджер, Администратор
- **Предусловие**: Пользователь авторизован, ТА существует
- **Основной сценарий**:
  1. Пользователь выбирает ТА и нажимает "Редактировать"
  2. Система загружает данные ТА
  3. Система открывает диалог редактирования с заполненными полями
  4. Пользователь изменяет необходимые поля
  5. Пользователь нажимает "Сохранить"
  6. Система валидирует данные
  7. Система обновляет запись в БД
  8. Система обновляет список
- **Постусловие**: Данные ТА обновлены

#### UC-11: Удалить торговый автомат
- **Акторы**: Администратор
- **Предусловие**: Пользователь авторизован как Администратор, ТА существует
- **Основной сценарий**:
  1. Пользователь выбирает ТА и нажимает "Удалить"
  2. Система показывает диалог подтверждения
  3. Пользователь подтверждает удаление
  4. Система удаляет ТА и связанные записи (каскадно)
  5. Система обновляет список
  6. Система показывает уведомление об успехе
- **Альтернативный сценарий**:
  - 3a. Пользователь отменяет: Диалог закрывается, ТА не удаляется
- **Постусловие**: Торговый автомат удалён

#### UC-12: Отвязать модем от торгового автомата
- **Акторы**: Инженер, Менеджер, Администратор
- **Предусловие**: ТА имеет привязанный модем (KitOnlineId не null)
- **Основной сценарий**:
  1. Пользователь выбирает ТА и нажимает "Отвязать модем"
  2. Система показывает диалог подтверждения
  3. Пользователь подтверждает
  4. Система устанавливает KitOnlineId = null
  5. Система обновляет список
- **Постусловие**: Модем отвязан, ТА офлайн

#### UC-20: Забронировать автомат

  8. Франчайзи подписывает договор (canvas с подписью)
  9. Система сохраняет договор

- **Включает**: Выбрать период аренды, Указать тип франчайзи, Подписать договор

**Входы:**
- Данные франчайзи (ФИО, тип, контакты)
- Выбранные ТА для аренды
- Электронная подпись (Canvas)

**Выходы:**
- Созданные договоры (таблица contracts)
- Audit Log (история действий)
- PDF документы договоров

**Управления:**
- Шаблоны договоров
- Правила аренды (стоимость, сроки)

**Механизмы:**
- Таблица contracts в БД
- rentalManager (Alpine.js компонент)
- Canvas API для электронной подписи
- LocalStorage для временного хранения
